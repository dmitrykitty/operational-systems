# Signals in UNIX / Linux

Signals are a lightweight mechanism used to **notify**, **interrupt**, or **control** a process.

They are asynchronous, which means they can arrive at almost any moment while the program is running.

Typical uses:

* stop a process
* terminate a process
* notify a process that an event happened
* wake up a process that is waiting
* react to `Ctrl + C`
* notify a parent that a child has finished

Signals are **not** a general-purpose data transfer mechanism like pipes or sockets.
They are mainly used for **events and control**.


## Common signals

Some important signals:

* `SIGINT` — interrupt, usually generated by `Ctrl + C`
* `SIGTERM` — polite request to terminate
* `SIGKILL` — immediate forced termination
* `SIGSTOP` — stop a process
* `SIGCONT` — continue a stopped process
* `SIGCHLD` — child process changed state
* `SIGUSR1`, `SIGUSR2` — user-defined signals for your own logic
* `SIGALRM` — timer/alarm signal
* `SIGSEGV` — invalid memory access

Important note:

* `SIGKILL` cannot be caught, ignored, or handled
* `SIGSTOP` also cannot be caught, ignored, or handled

---

# Main functions related to signals

The most important functions in this topic are:

* `signal()` — simple way to install a signal handler
* `sigaction()` — more robust and flexible way to install a signal handler
* `kill()` — send a signal to another process
* `raise()` — send a signal to the current process itself
* `pause()` — suspend execution until a signal arrives

---

# `signal()`

`signal()` is the old and simple way to install a signal handler.

A signal handler is a function that will be executed when a given signal arrives.

## Syntax

```c
void (*signal(int signum, void (*handler)(int)))(int);
```

You usually use it like this:

```c
signal(SIGUSR1, handler);
```

Meaning:

* when `SIGUSR1` arrives
* run the function `handler`

## Example

```c
#include <stdio.h>
#include <signal.h>
#include <unistd.h>

void handler(int sig) {
    printf("Received signal %d\n", sig);
}

int main() {
    signal(SIGUSR1, handler);

    while (1) {
        pause();
    }

    return 0;
}
```

## What this does

* installs a handler for `SIGUSR1`
* process then waits in `pause()`
* when `SIGUSR1` is received, `handler()` is executed

## Why `signal()` is not ideal

`signal()` is simple, but:

* historically its behavior was not perfectly consistent across systems
* it gives less control
* it does not expose advanced options

Because of that, modern code usually prefers `sigaction()`.

---

# `kill()`

`kill()` is used to **send a signal to another process**.

Despite the name, it does not only kill processes.

## Syntax

```c
int kill(pid_t pid, int sig);
```

## Example

```c
kill(pid, SIGTERM);
```

Meaning:

* send `SIGTERM` to process `pid`

## More examples

```c
kill(pid, SIGUSR1);   // send user-defined signal
kill(pid, SIGSTOP);   // stop the process
kill(pid, SIGCONT);   // continue the process
kill(pid, SIGKILL);   // forcefully terminate
```

## What `kill()` actually does

It only sends a signal.

What happens next depends on:

* the signal type
* whether the process installed a handler
* whether the signal can be caught or ignored
* the default action for that signal

---

# `raise()`

`raise()` sends a signal to the **current process itself**.

## Syntax

```c
int raise(int sig);
```

## Example

```c
raise(SIGUSR1);
```

Meaning:

* current process sends `SIGUSR1` to itself

This is useful for:

* testing handlers
* intentionally triggering signal-based logic
* simple demonstrations

## Example

```c
#include <stdio.h>
#include <signal.h>
#include <stdlib.h>

void handler(int sig) {
    printf("Process received signal %d from itself\n", sig);
}

int main() {
    signal(SIGUSR1, handler);
    raise(SIGUSR1);
    return 0;
}
```

---

# `pause()`

`pause()` suspends the process until a signal is delivered.

## Syntax

```c
int pause(void);
```

## Example

```c
pause();
```

Meaning:

* stop execution
* wait until a signal arrives

When a signal is handled, `pause()` returns.

Usually:

* return value is `-1`
* `errno` is set to `EINTR`

## Example

```c
#include <stdio.h>
#include <signal.h>
#include <unistd.h>

void handler(int sig) {
    printf("Signal received: %d\n", sig);
}

int main() {
    signal(SIGUSR1, handler);

    printf("Waiting for a signal...\n");
    pause();

    printf("pause() returned\n");
    return 0;
}
```

## What to remember

`pause()` does not wait for a specific signal.
It wakes up when **any non-ignored signal** is delivered.

---

# `sigaction()` — the most important one

`sigaction()` is the modern and preferred way to define how a process should react to signals.

Compared to `signal()`, it is:

* more explicit
* more flexible
* more reliable
* more portable

---

## Why `sigaction()` exists

`signal()` is too simple for serious control.

Sometimes you need to decide:

* which handler to use
* which other signals should be blocked while the handler runs
* whether interrupted system calls should be automatically restarted
* whether you want more information about the signal sender
* whether the signal should behave differently for special cases

`sigaction()` gives you that control.

---

# Basic syntax of `sigaction()`

```c
int sigaction(int signum, const struct sigaction *act, struct sigaction *oldact);
```

Parameters:

* `signum` — which signal you are configuring
* `act` — pointer to the new signal action
* `oldact` — optional pointer where previous action is stored

Typical call:

```c
sigaction(SIGUSR1, &sa, NULL);
```

Meaning:

* set behavior for `SIGUSR1`
* use configuration stored in `sa`
* do not save previous configuration

---

# `struct sigaction`

This is the key structure.

A simplified view:

```c
struct sigaction {
    void     (*sa_handler)(int);
    void     (*sa_sigaction)(int, siginfo_t *, void *);
    sigset_t   sa_mask;
    int        sa_flags;
};
```

Depending on the system, the exact low-level definition may vary a bit, but conceptually these are the important fields.

---

# Important fields of `struct sigaction`

## 1. `sa_handler`

Used for the simple handler form.

Example:

```c
void handler(int sig) {
    printf("Handled signal %d\n", sig);
}
```

Then:

```c
sa.sa_handler = handler;
```

Meaning:

* when the signal arrives, call `handler(int sig)`

---

## 2. `sa_sigaction`

Used for the advanced handler form.

It gives extra information, such as:

* which PID sent the signal
* more signal metadata

The function has this form:

```c
void handler(int sig, siginfo_t *info, void *ucontext);
```

If you want to use `sa_sigaction`, you must usually set:

```c
sa.sa_flags = SA_SIGINFO;
```

Then the kernel knows to use the 3-argument handler.

---

## 3. `sa_mask`

This is a set of signals that should be **blocked while the handler is running**.

That means:

* if your handler is currently executing
* signals from `sa_mask` are temporarily blocked
* this helps avoid handler reentry or interference

You usually initialize it with:

```c
sigemptyset(&sa.sa_mask);
```

Meaning:

* do not block any extra signals while handler runs

Or you can add signals:

```c
sigaddset(&sa.sa_mask, SIGUSR2);
```

Meaning:

* while handling this signal, also block `SIGUSR2`

---

## 4. `sa_flags`

Flags modify handler behavior.

Common flags:

* `0` — basic/default behavior
* `SA_SIGINFO` — use `sa_sigaction` instead of `sa_handler`
* `SA_RESTART` — restart certain interrupted system calls automatically
* `SA_NOCLDSTOP` — with `SIGCHLD`, do not notify when children stop
* `SA_RESETHAND` — reset handler to default after first signal

The most important for beginners are:

* `0`
* `SA_SIGINFO`
* `SA_RESTART`

---

# Why people write `memset(&sa, 0, sizeof(sa))`

This is the part that often confuses people.

Example:

```c
struct sigaction sa;
memset(&sa, 0, sizeof(sa));
```

## What it means

* `sa` is a structure
* a structure may contain many fields
* if you do not initialize them, some fields may contain garbage values
* `memset(..., 0, ...)` fills the whole structure with zeros

So after that:

* all bytes inside `sa` are zero
* pointer fields become `NULL`
* numeric fields become `0`
* masks start in a clean state

This is a common “clean initialization” pattern.

## Why it is useful

Because then you only set the fields you care about:

```c
sa.sa_handler = handler;
sigemptyset(&sa.sa_mask);
sa.sa_flags = 0;
```

Without `memset()`, you could accidentally leave some field uninitialized.

---

# What about `sigemptyset()`?

You often see both:

```c
memset(&sa, 0, sizeof(sa));
sigemptyset(&sa.sa_mask);
```

and it can look repetitive.

## Why both?

Because `sa_mask` is of type `sigset_t`, and the official, correct API to initialize signal sets is:

```c
sigemptyset(&sa.sa_mask);
```

Even if the structure was already zeroed, using `sigemptyset()` is clearer and semantically correct.

So:

* `memset()` → clear whole structure
* `sigemptyset()` → explicitly initialize signal mask as an empty set

That is why people often do both.

---

# Minimal `sigaction()` example

```c
#include <stdio.h>
#include <stdlib.h>
#include <signal.h>
#include <string.h>
#include <unistd.h>

void handler(int sig) {
    printf("Handled signal %d\n", sig);
}

int main() {
    struct sigaction sa;

    memset(&sa, 0, sizeof(sa));
    sa.sa_handler = handler;
    sigemptyset(&sa.sa_mask);
    sa.sa_flags = 0;

    if (sigaction(SIGUSR1, &sa, NULL) == -1) {
        perror("sigaction");
        exit(1);
    }

    printf("PID: %d\n", getpid());

    while (1) {
        pause();
    }

    return 0;
}
```

## What happens here

1. create `struct sigaction sa`
2. clear it with `memset`
3. set handler function
4. initialize empty mask
5. use no special flags
6. install behavior for `SIGUSR1`
7. wait in `pause()`

---

# Advanced `sigaction()` example with sender PID

Now the more interesting version:

```c
#include <stdio.h>
#include <stdlib.h>
#include <signal.h>
#include <string.h>
#include <unistd.h>

void handler(int sig, siginfo_t *info, void *ucontext) {
    printf("Received signal %d from PID %d\n", sig, info->si_pid);
}

int main() {
    struct sigaction sa;

    memset(&sa, 0, sizeof(sa));
    sa.sa_sigaction = handler;
    sigemptyset(&sa.sa_mask);
    sa.sa_flags = SA_SIGINFO;

    if (sigaction(SIGUSR1, &sa, NULL) == -1) {
        perror("sigaction");
        exit(1);
    }

    printf("PID: %d\n", getpid());

    while (1) {
        pause();
    }

    return 0;
}
```

## Important difference

Here:

```c
sa.sa_sigaction = handler;
sa.sa_flags = SA_SIGINFO;
```

This means:

* use advanced 3-argument handler
* make sender information available

Now if another process sends `SIGUSR1` with `kill(pid, SIGUSR1)`, the receiver can know who sent it.

This is very useful for process-to-process signaling.

# Sending signals between two of your own processes

The most common approach:

* create child with `fork()`
* parent knows child PID from `fork()`
* child knows parent PID from `getppid()`
* one sends signals using `kill()`
* other waits with `pause()` and handles them with `sigaction()`

---

## Example: parent sends signal to child

```c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <signal.h>
#include <string.h>
#include <sys/wait.h>

volatile sig_atomic_t got_usr1 = 0;

void handler(int sig) {
    got_usr1 = 1;
}

int main() {
    struct sigaction sa;
    memset(&sa, 0, sizeof(sa));
    sa.sa_handler = handler;
    sigemptyset(&sa.sa_mask);
    sa.sa_flags = 0;

    pid_t rc = fork();
    if (rc < 0) {
        perror("fork");
        exit(1);
    }

    if (rc == 0) {
        if (sigaction(SIGUSR1, &sa, NULL) == -1) {
            perror("sigaction");
            _exit(1);
        }

        printf("Child PID: %d\n", getpid());

        while (1) {
            pause();

            if (got_usr1) {
                got_usr1 = 0;
                printf("Child received SIGUSR1\n");
                break;
            }
        }

        _exit(0);
    } else {
        sleep(1); // let child install handler
        kill(rc, SIGUSR1);
        wait(NULL);
    }

    return 0;
}
```

---

# Why use `volatile sig_atomic_t`?

In signal-related code, this is the safest simple type for communication between:

* asynchronous signal handler
* normal main code

Example:

```c
volatile sig_atomic_t got_signal = 0;
```

Then in handler:

```c
got_signal = 1;
```

This is better than doing complex logic inside the handler.

---

# Why handlers should be small

Signal handlers run asynchronously, which means they can interrupt normal code at almost any point.

Because of that, handlers should be minimal.

Good pattern:

* handler sets a flag
* main loop checks flag and performs real logic

Example:

```c
volatile sig_atomic_t got_usr1 = 0;

void handler(int sig) {
    got_usr1 = 1;
}
```

Then later in main:

```c
if (got_usr1) {
    got_usr1 = 0;
    printf("Signal noticed in main loop\n");
}
```

This is usually much safer than doing too much inside the handler itself.

---

# `signal()` vs `sigaction()` vs `kill()` — exact difference

This is the most important comparison.

## `kill()`

Purpose:

* **send** a signal to a process

It is about the **sender**.

Example:

```c
kill(pid, SIGUSR1);
```

---

## `signal()`

Purpose:

* define a simple handler for a signal

It is about the **receiver behavior**.

Example:

```c
signal(SIGUSR1, handler);
```

---

## `sigaction()`

Purpose:

* define signal handling in a more precise and configurable way

It is also about the **receiver behavior**, but much more advanced.

Example:

```c
sigaction(SIGUSR1, &sa, NULL);
```

---

## Short summary

* `kill()` → send signal
* `signal()` → simple signal handler
* `sigaction()` → robust configurable signal handler

---

# Is there more than `pause()`?

Yes.

Signals have a bigger ecosystem.

Important related functions include:

* `alarm()` — send `SIGALRM` after some time
* `sigprocmask()` — block / unblock signals
* `sigsuspend()` — wait for signals with temporary mask
* `sigwait()` — synchronously wait for signals
* `killpg()` — send signal to a process group

But for first understanding, the core set is:

* `signal()`
* `sigaction()`
* `kill()`
* `raise()`
* `pause()`

---

# Example: `raise()` + `sigaction()`

```c
#include <stdio.h>
#include <stdlib.h>
#include <signal.h>
#include <string.h>

void handler(int sig) {
    printf("Handled signal %d\n", sig);
}

int main() {
    struct sigaction sa;

    memset(&sa, 0, sizeof(sa));
    sa.sa_handler = handler;
    sigemptyset(&sa.sa_mask);
    sa.sa_flags = 0;

    if (sigaction(SIGUSR1, &sa, NULL) == -1) {
        perror("sigaction");
        exit(1);
    }

    raise(SIGUSR1);

    return 0;
}
```

This shows:

* `sigaction()` defines what happens when `SIGUSR1` arrives
* `raise()` sends `SIGUSR1` to the current process
* handler gets executed

# Typical beginner mistakes

## 1. Forgetting to initialize `struct sigaction`

Bad:

```c
struct sigaction sa;
sa.sa_handler = handler;
sigaction(SIGUSR1, &sa, NULL);
```

Why bad:

* some fields may contain garbage

Better:

```c
struct sigaction sa;
memset(&sa, 0, sizeof(sa));
sa.sa_handler = handler;
sigemptyset(&sa.sa_mask);
sa.sa_flags = 0;
```

---

## 2. Mixing `sa_handler` and `sa_sigaction`

Use one or the other depending on your design.

* simple handler → `sa_handler`
* advanced 3-argument handler → `sa_sigaction` + `SA_SIGINFO`

---

## 3. Doing too much in the handler

Handlers should not become mini-programs.

Prefer:

* set a flag
* do real work in main loop

---

## 4. Forgetting that `pause()` wakes up on any signal

`pause()` does not mean:

* “wait for only SIGUSR1”

It means:

* “sleep until some signal arrives”

---

## 5. Thinking `kill()` always kills

It does not.
It just sends a signal.

---

# Recommended rule

For simple toy examples:

* `signal()` is acceptable

For anything more serious:

* prefer `sigaction()`

This is the practical rule most people follow.

---

# Interview-style explanation

You can describe it like this:

`signal()` and `sigaction()` are used to define how a process reacts to a signal, while `kill()` is used to send a signal to a process. `signal()` is the older and simpler interface, but `sigaction()` is preferred because it gives more control through fields like `sa_mask` and `sa_flags`, and can also provide extra information with `SA_SIGINFO`. `pause()` is used to suspend a process until a signal arrives, and `raise()` is used when a process wants to send a signal to itself. In practice, `sigaction()` is the standard way to install handlers in robust code.

---

# Very short summary

* signal = asynchronous notification
* `kill()` = send signal to another process
* `raise()` = send signal to yourself
* `pause()` = sleep until a signal arrives
* `signal()` = simple handler installation
* `sigaction()` = better, modern, configurable handler installation
